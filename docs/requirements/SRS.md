# 1 Введение

<a name="appointment"/>

## 1.1 Назначение
В этом документе описаны функциональные и нефункциональные требования к веб-платформе «ResourceHub». Документ предназначен для команды разработки, тестирования и управления проектом с целью реализации и проверки корректности работы приложения.

<a name="business_requirements"/>

## 1.2 Бизнес-требования

<a name="initial_data"/>

### 1.2.1 Исходные данные
Интернет насыщен полезными материалами, но пользователям сложно систематизировать качественные ресурсы, делиться ими и находить новые. Существующие решения или слишком просты (букмарки), или перегружены лишним функционалом. Не хватает удобной платформы для обмена и структурирования ссылок и материалов по интересам.

<a name="business_opportunities"/>

### 1.2.2 Возможности бизнеса
ResourceHub предоставляет единое пространство, где пользователи могут сохранять, классифицировать и рекомендовать полезные ресурсы: статьи, видео, инструменты, курсы. Возможность создавать коллекции и подписываться на других пользователей формирует сообщество, стимулирующее обмен знаниями. Сервис может монетизироваться через подписку на премиум-функции (расширенная аналитика, приватные коллекции) и партнёрские ссылки.

<a name="project_boundary"/>

### 1.2.3 Границы проекта
Проект «ResourceHub» включает в себя публичный сайт с лентой ресурсов, личный кабинет пользователя, систему голосований и комментариев, а также административную панель для модерации. В первую версию не входит интеграция с платёжными системами, сложные алгоритмы персональных рекомендаций и мобильные приложения.

<a name="analogues"/>

## 1.3 Аналоги
Аналогами можно считать сервисы вроде **Reddit**, **Pinterest**, **Hacker News**. Они позволяют делиться ссылками, но не предоставляют удобных средств создания тематических коллекций и подписок на них. ResourceHub делает акцент на структурирование контента и качественные рекомендации.

<a name="user_specifications"/>

## 2.3 Характеристики пользователей

<a name="user_classes"/>

### 2.3.1 Классы пользователей

| Класс пользователей | Описание |
|:---|:---|
| **Администратор** | Пользователь с расширенными правами для модерации контента и управления пользователями. Может удалять материалы, блокировать аккаунты, просматривать отчёты о нарушениях. |
| **Зарегистрированный пользователь** | Основной участник платформы. Может добавлять ресурсы, создавать коллекции, голосовать, комментировать, подписываться на другие коллекции и получать уведомления. |
| **Гость** | Нерегистрированный посетитель. Имеет доступ к просмотру публичных ресурсов и коллекций, но не может создавать контент и взаимодействовать с ним (нет возможности комментировать, голосовать, подписываться). |

<a name="application_audience"/>

### 2.3.2 Аудитория приложения
* Студенты, разработчики, исследователи, профессионалы разных сфер, которые ищут и сохраняют полезные ссылки, статьи, курсы, инструменты.  
* Люди, желающие делиться полезными материалами и находить единомышленников.  
* Сообщества и небольшие команды, которым нужен единый каталог ресурсов для совместной работы.

<a name="assumptions_and_dependencies"/>

## 2.4 Предположения и зависимости
1. Для использования функций добавления ресурсов, голосования и подписки требуется регистрация.
2. Все основные действия (создание/редактирование/комментирование) требуют стабильного интернет-соединения.
3. В первой версии нет интеграции с платёжными системами и внешними API рекомендаций.

<a name="system_requirements"/>

# 3 Системные требования

<a name="functional_requirements"/>

## 3.1 Функциональные требования

<a name="admin_main_functions"/>

### 3.1.1 Основные функции Администратора

<a name="admin_auth"/>

#### 3.1.1.1 Аутентификация и управление аккаунтом Администратора
**Описание.** Администратор должен иметь возможность безопасно входить в панель модерации.

| Функция | Требования |
|:---|:---|
| Регистрация администратора | Создание аккаунта только через внутренние инструменты или вручную через базу данных. |
| Вход в панель | Авторизация по логину и паролю, хранение пароля в хешированном виде. |
| Управление пользователями | Блокировка/разблокировка аккаунтов, просмотр отчетов о нарушениях. |

<a name="resource_management"/>

#### 3.1.1.2 Управление ресурсами и коллекциями
| Функция | Требования |
|:---|:---|
| Удаление ресурса | Возможность удалять нежелательный контент. |
| Модерация комментариев | Возможность скрывать или удалять комментарии, нарушающие правила. |

<a name="content_moderation"/>

#### 3.1.1.3 Модерация контента
| Функция | Требования |
|:---|:---|
| Просмотр отчетов | Просмотр списка жалоб на ресурсы и пользователей. |
| Принятие мер | Возможность вынести предупреждение или заблокировать пользователя. |

<a name="user_main_functions"/>

### 3.1.2 Основные функции Пользователя

<a name="user_auth"/>

#### 3.1.2.1 Аутентификация и профиль
| Функция | Требования |
|:---|:---|
| Регистрация | Создание аккаунта с подтверждением email. |
| Вход | Авторизация по email и паролю. |
| Редактирование профиля | Изменение имени, аватара, описания. |

<a name="resource_usage"/>

#### 3.1.2.2 Работа с ресурсами
| Функция | Требования |
|:---|:---|
| Добавление ресурса | Ввод заголовка, описания, URL, выбор коллекции. |
| Создание коллекции | Создание и редактирование собственных подборок ресурсов. |
| Подписка | Возможность подписываться на чужие коллекции. |
| Голосование и комментарии | Лайк/дизлайк ресурсов, добавление комментариев. |

<a name="restrictions_and_exclusions"/>

### 3.1.3 Ограничения и исключения
1. Только Администратор может удалять или блокировать контент.
2. Гость может только просматривать публичные ресурсы.
3. В первой версии нет личных сообщений и чатов.

<a name="non-functional_requirements"/>

## 3.2 Нефункциональные требования

<a name="quality_attributes"/>

### 3.2.1 Атрибуты качества

<a name="requirements_for_ease_of_use"/>

#### 3.2.1.1 Требования к удобству использования
1. Интерфейс должен быть адаптивным и простым для понимания без обучения.
2. Все ключевые действия доступны максимум в 2–3 клика.

<a name="security_requirements"/>

#### 3.2.1.2 Требования к безопасности
1. Пароли хранятся в хешированном виде.
2. Все чувствительные данные передаются по HTTPS.
3. Данные пользователей изолированы.

<a name="access_requirements"/>

#### 3.2.1.3 Требования к производительности и доступности
1. Время отклика сервера не более 1–2 секунд.
2. Доступность сервиса не ниже 99%.
3. Масштабируемость для роста числа пользователей.

<a name="external_interfaces"/>

### 3.2.2 Внешние интерфейсы
* Поддержка последних версий Chrome, Firefox, Safari, Edge.
* Корректная работа на десктопах, планшетах, смартфонах.

<a name="restrictions"/>


### 3.2.3 Ограничения
1. Веб-приложение реализуется на **Spring Boot** (Java) как backend-сервис с REST API.
2. В качестве базы данных используется **PostgreSQL**.
3. Frontend может быть любым SPA-клиентом (например, React/TypeScript), который взаимодействует с REST API.
4. Язык разработки backend: **Java 17+**.
